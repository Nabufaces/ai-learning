# Week 1: LLM API 基础 + 第一个对话 CLI

## 本周目标

本周的目标不是掌握所有模型参数，而是先建立“AI 应用是一个软件系统”的基本直觉：一次 LLM 调用包含输入构造、模型选择、上下文限制、输出解析、错误处理、成本控制和可重复验证。你需要先跑通一个最小 CLI，再理解为什么后续要做 schema、RAG、tool calling 和 eval。

## 核心概念详解

### Token 和上下文窗口

Token 是模型处理文本的基本单位。中文、英文、符号和空格都会被切成 token，但 token 不等于字或字符。上下文窗口是模型一次请求中能看到的最大 token 数，包括系统提示、用户问题、检索材料、工具结果和模型即将生成的输出。

工程上最重要的结论是：上下文窗口不是无限记忆。长文档、长对话和大量工具结果不能无脑塞进 prompt。后续 Week 3 的 context 管理和 Week 4 的 RAG 都是在解决“哪些信息应该进入本次上下文”的问题。

### Prompt 的角色

一个最小 AI 应用通常会有几类提示：

- System/developer 指令：定义助手身份、边界、输出要求。
- User 输入：用户当前要解决的问题。
- Retrieved context：从文档、数据库或工具取回的资料。
- Tool result：外部工具执行结果。

不要把所有内容都混成一个大 prompt。不同来源的信息可信度不同，尤其是 retrieved context 可能包含恶意指令，后续 Week 7 会把它当作 prompt injection 风险处理。

### 模型参数

- `temperature` 控制随机性。学习、检索问答、结构化输出通常用较低值。
- `max tokens` 控制最大输出长度。过小会截断，过大会浪费成本。
- `model` 决定能力、延迟、价格和上下文长度。
- `stop sequence` 可以用于截断特定格式，但在现代结构化输出里不是首选方案。

参数不是“调到最好”的魔法旋钮。更可靠的做法是先定义任务 contract，再用 eval 看参数变化是否改善结果。

### 结构化输出

AI 应用经常需要模型返回 JSON、枚举、字段或数组。只在 prompt 里写“请返回 JSON”不够，因为模型可能漏字段、类型错误或夹带解释文本。应用侧必须做 runtime validation，失败后选择重试、降级或请求人工确认。

本周的 `pnpm lab:structured` 用手写校验演示这个思想。Week 3 会进一步引入 Zod。

## 主项目增量

本周还不要求改 AI 学习助手主项目，只需要跑通 CLI 和结构化输出 lab。这样 Week 2 上 Web/API 时，你已经知道一次模型调用的最小闭环是什么。

## 动手任务

1. 运行 `pnpm lab:hello`，观察 mock mode 输出。
2. 可选：设置 `OPENAI_API_KEY` 后调用真实 API。
3. 运行 `pnpm lab:structured`。
4. 修改 structured lab 的样例，制造缺字段、类型错误、空数组三种失败。
5. 写一段笔记：输出格式错误和事实错误分别应该怎么处理。

## 验收标准

- 能说清楚一次 API 请求的输入、输出、错误和 fallback。
- 能解释 token、上下文窗口和 max output 的关系。
- 能解释为什么 AI 应用需要 schema，而不是只靠 prompt。
- 能给出结构化输出失败后的恢复策略。

## 常见误区

- 误区：先学很多 prompt 技巧，再写代码。  
  更好的顺序：先跑通 API，再把 prompt 当作可测试的软件接口。

- 误区：模型输出看起来像 JSON 就可以直接用。  
  更好的做法：所有外部输入都需要 validation，模型输出也是外部输入。

## 复盘问题

- 哪些问题应该靠模型解决，哪些应该靠程序验证？
- 当模型输出不符合 schema 时，应该重试、降级还是请求人工确认？
- 如果没有 eval，你如何知道换模型后结果变好了还是变差了？
